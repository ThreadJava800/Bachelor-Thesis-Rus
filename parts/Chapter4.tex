\section{Реализация механизма сбора диагностической информации}
\label{sec:Chapter4} \index{Chapter4}

Разработанный механизм сбора диагностической информации включает
три основных компонента: \\
1. Виртуальное устройства на стороне гипервизора Stratovirt. \\
2. Драйвер для среды UEFI. \\
3. Драйвер для операционной системы Windows.

Каналом взаимодействия между данными компонентами служит кольцевой буфер,
расположенный в общей памяти, разделяемой между гостевой ОС и гипервизором.

Виртуальное устройство эмулируется в гипервизоре Stratovirt и выступает в
роли единой точки сбора диагностической информации. Таким образом, основными
функциями устройства являются: \\ 
1. Приём диагностических сообщений от драйверов гостевых систем. \\
2. Сериализация сообщений и запись их в файл. \\
3. Хранение метаинформации о буфере: его размере и типе поведения при
переполнении. \\
4. Предоставление доступа к метаинформации посредством регистров виртуального
устройства.

Драйвер для среды UEFI выполняет следующие задачи: \\
1. Выделение буфера в гостевой памяти типа runtime: доступной
после окончания работы UEFI (после вызова ExitBootServices) и во время работы
операционной системы. \\
2. Предоставление интерфейса для записи диагностических сообщения в буфер. \\
3. Регистрация обработчиков событий ExitBootServices и SetVirtualAddressMap
для обеспечения работы драйвера с EfiRuntimeService.

Для упрощения интеграции с существующими UEFI модулями была разработана
собственная версия DebugLib, переопределяющая стандартные функции логирования.
Эти функции обращаются к вышеупомянутому драйверу для передачи сообщений.

Для обеспечения взаимодействия компонентов гостевой системы с буфером
реализован драйвер, позволяющий: \\
1. User-space приложениям и сервисам взаимодействовать с буфером. \\
2. Kernel-space драйверам взаимодействовать с буфером.

Виртуальное устройство может быть добавлено в устройство при помощи флага
-logdevice, который принимает следующие параметры: id, flush\_logs, iothread: \\
1. id - идентификатор виртуального устройства, позволяющий однозначно
определить его в системе. \\
2. flush\_logs - параметр логического типа, который регламентирует поведение при
переполнении буфера. Если параметр установлен в true, запись сообщения будет
ожидать освобождение места в буфере. Если false - сообщение, не поместившееся в буфер
будет пропущено. \\
3. iothread - идентификатор контекста, в котором работает поток-читатель. Параметр
может быть пропущен --- в таком случае поток-читатель будет работать в главном контексте
(main loop).

Виртуальное устройство взаимодействует с гостевыми операционными системами
через регистров, доступ к которым осуществляется посредством MMIO (memory-mapped I/O).
В частности, среди прочих существует регистр, позволяющий получить информацию о
поведении при переполнении буфера.

Как уже отмечалось, выделение памяти буфера выполняется драйвером UEFI.
Однако на ранних стадиях работы прошивки (SEC и PEI) эта операция
невозможна, т.к. память ещё не инициализирована. По этой причине,
на этих этапах требуется альтернативный способ передачи диагностических
данных в виртуальное устройство, поэтому в него был добавлен специальный
регистр, позволяющий записываются данные последовательно ---
по 8, 4, 2 байта или по одному в зависимости от размера сообщения.
Виртуальное устройство накапливает поступающие байты до теъ пор,
пока не встретит символ окончания строки '\textbackslash 0', после чего сформированное
сообщение записывается в выходной файл. 

После инициализации памяти (на этапе PEIM) появляется возможность выделить буфер.
Дальнейшая передача сообщений происходит уже через этот буфер.

Общая память организована в виде неблокирующего (lock-free) кольцевого буфера с
несколькими писателями и одним читателем (MPSC --- Multiple Producer Single Consumer).
В начале буфера располагается заголовок, содержащий указатели текущих позиций читателя
и писателя, флаг переполнения, а также вспомогательную структуру SHADOW\_POS ("мнимая позиция").
Данная структура включает два поля типа UINT32: счётчик активных писателей,
выполняющих копирование сообщений, и позицию резервирование места для новых сообщений.

Структура буфера представлена ниже:

\begin{lstlisting}
    typedef union {
        struct {
            uint32_t ReservedPos;
            uint32_t WriterCnt;
        } S;
        uint64_t U;
    } SHADOW_W_POS;

    typedef struct {
        uint32_t ReadPos;
        uint32_t WritePos;
        uint64_t IsOverflow;
        SHADOW_W_POS ShWPos;
        char Buffer[0]; 
    } LOCKLESS_RING_BUFFER;
\end{lstlisting}

Процедура записи в буфер выполняется в три этапа: резервирование места для сообщения
(reserve), копирование в забронированную область (copy) и оповещение об окончании
копирования (commit).

Сначала каждый поток пытается зарезервировать место для своего сообщения.
Для этого каждый поток загружает текущее значение SHADOW\_POS и с помощью
атомарной операции compare\_exchange пытается обновить его в заголовке буфера,
увеличивая счётчик писателей и смещая позицию записи на размер сообщения.
В случае конфликта (если compare\_exchange вернула значение false) операция
повторяется.

Таким образом, гарантируется, что никакой другой поток не сможет зарезервировать
себе то же самое место: каждый поток копирует своё сообщение в уникальное область
буфера.

Псевдокод данного этапа представлен ниже:

\begin{lstlisting}
    uint32_t LrbReserve (
        LOCKLESS_RING_BUFFER *Rb,
        const uint32_t MsgSize,
        SHADOW_W_POS *Curr
    ) {
        for (;;) {
            Curr->U = Rb->ShWPos.U;
            SHADOW_W_POS NextShadowPos = {
                .S.ReservedPos = Curr->S.ReservedPos + MsgSize,
                .S.WritePos = Curr->S.WriterCnt + 1
            };

            if (atomic_compare_exchange(&Rb->ShWPos, &Curr->U, Next.U)) {
                // SHADOW_W_POS была обновлена в буфере
                // Таким образом, гарантируется, что ни один другой поток
                // не будет копировать своё сообщение в занятую память.
                return Curr->S.ReservedPos;
            }
        }
    }
\end{lstlisting}

После того, как поток скопировал сообщение, он сообщает об окончании операции.
Поток атомарно уменьшает счётчик активных писателей в SHADOW\_POS, используя
аналогичный резервированию механизм синхронизации. Как только количество писателей
в SHADOW\_POS становится равным нулю, обновляется глобальный указатель писателя в
заголовке буфера. Таким образом, указатель писателя обновляется после завершения
всех операций записи, что гарантирует целостность данных в буфере и исключает
чтение неполных записей на читающей стороне.

Псевдокод алгоритма представлен ниже:

\begin{lstlisting}
    uint32_t LrbCommit (
        LOCKLESS_RING_BUFFER *Rb,
        SHADOW_W_POS *Curr
    ) {
        for (;;) {
            SHADOW_W_POS NextShadowPos = {
                .S.ReservedPos = Curr->S.ReservedPos,
                .S.WritePos = Curr->S.WriterCnt - 1
            };

            if (NextShadowPos.S.WriterCnt == 0) {
                // все сообщения до текущего закончили копирование
                Rb->WritePos = Curr->S.ReservedPos;
            }
            if (atomic_compare_exchange(&Rb->ShWPos, &Curr->U, Next.U)) {
                return;
            }
        }
    }
\end{lstlisting}

После завершения записи драйвер уведомляет виртуальное устройство через отдельный
регистр о том, что глобальная позиция писателя изменилась и сообщения готовы к чтению.

Функции записи также должны проводить некоторые проверки буфера, чтобы предотвратить
переполнение и запись в повреждённый буфер. Наивный алгоритм проверки мог бы выглядеть
следующим образом:

\begin{lstlisting}
    uint32_t LrbReserve (
        LOCKLESS_RING_BUFFER *Rb,
        const uint32_t MsgSize,
        SHADOW_W_POS *Curr
    ) {
        for (;;) {
            Curr->U = Rb->ShWPos.U;
            uint32_t ReadPos = Rb->ReadPos;
            uint32_t BusySize = Curr->S.ReservedPos - ReadPos;
            if (BusySize > BUFFER_SIZE) {
                // Буфер сломан:
                // размер занятой области не может быть больше всего буфера. 
                return -1;
            }
            if (BusySize + Size > BUFFER_SIZE) {
                // Сообщение не поместилось.
                // В зависимости от выбранного поведения, сообщение либо
                // пропускается, либо ожидает завершения записи других.
            }

            ...
        }
    }
\end{lstlisting}

Однако при таком подходе возникает риск ложно принять буфер за повреждённый, если
между операциями чтения структуры SHADOW\_POS и проверкой целостности буфера другой
поток успеет зарезервировать место в буфере, скопировать сообщение и выполнить
операцию commit. Рассмотрим это на примере:

Пусть изначально: \\
1. Позиция читателя ReadPos = 100. \\
2. Аисателя WritePos = 200. \\
3. Количество активных писателей ShWPos.WriterCnt = 1.

При описанном ниже планировании потоков возникает указанная проблема.

Таким образом, переменная BusySize переполняется, и буфер ложно
считается повреждённым.

Данную проблема решается повторной загрузкой значений Curr->U и ReadPos.
Если Curr->U не изменилось, то буфер действительно повреждён.
В противном случае проверку необходимо повторить.

Псевдокод проверки представлен ниже:

\begin{lstlisting}
    uint32_t LrbReserve (
        LOCKLESS_RING_BUFFER *Rb,
        const uint32_t MsgSize,
        SHADOW_W_POS *Curr
    ) {
        for (;;) {
            Curr->U = Rb->ShWPos.U;
            uint32_t ReadPos = Rb->ReadPos;
            uint32_t BusySize = Curr->S.ReservedPos - ReadPos;

            while (BusySize > BUFFER_SIZE) {
                uint64_t CurrCpy = Curr->U;

                // Принудительно перечитать значения ReadPos и ShWPos
                MemoryFence();
                Curr->U = Rb->ShWPos.U;
                ReadPos = Rb->ReadPos;

                if (Curr->U == CurrCpy) {
                    return -1;
                }
                BusySize = Curr->S.ReservedPos - ReadPos;
            }
            if (BusySize + Size > BUFFER_SIZE) {
                // Сообщение не поместилось.
                // В зависимости от выбранного поведения, сообщение либо
                // пропускается, либо ожидает завершения записи других.
            }

            ...
        }
    }
\end{lstlisting}

Алгоритм потока-читателя ничем в сущности не отличается от чтения из
обычного кольцевого буфера. При получении уведомления об изменении
позиции писателя через регистр виртуального устройства, поток-читатель
последовательно извлекает сообщения, начиная с позиции ReadPos и
продвигаясь к WritePos, пока не будут прочитаны все доступные данные.

Вышеописанный алгоритм читателя реализован как в драйвере UEFI, так и
в драйвере для Windows.

В драйвере для Windows были реализованы ioctl -
команды для взаимодействия с буфером: \\
1. Для пользовательского программ (user-space): она принимает сообщение и
выполняет проверки над ним и записывает его в буфер. \\
2. Для программ режима ядра (kernel-space) --- возвращает указатель
на функцию записи в буфер.
